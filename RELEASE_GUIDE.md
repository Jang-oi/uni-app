# GitHub Releases 작성 가이드

uni-app의 릴리즈 메시지는 **간단명료**하게 작성합니다.

## 릴리즈 생성 방법

### 1. GitHub에서 릴리즈 생성

1. GitHub 저장소 페이지로 이동
2. 오른쪽 사이드바에서 `Releases` 클릭
3. `Create a new release` 버튼 클릭

### 2. 버전 태그 규칙

- **형식**: `v{major}.{minor}.{patch}`
- **예시**: `v1.2.3`, `v2.0.0`

**버전 타입**:

- `v1.0.0` → Major (대규모 변경, 새로운 기능)
- `v1.1.0` → Minor (기능 추가, 개선)
- `v1.1.1` → Patch (버그 수정, 작은 개선)

### 3. 릴리즈 메시지 작성

#### ✅ 권장 형식

각 변경사항을 **한 줄**로 작성하고, `-`로 시작:

```markdown
- 휴가 달력 UI/UX 대폭 개선
- 자동 업데이트 기능 추가
- 업무 테이블 필터링 버그 수정
- HyperV 모니터링 성능 최적화
```

#### ✅ 좋은 예시

```markdown
- IPC 핸들러 모듈화로 코드 구조 개선
- GitHub Releases 연동으로 자동 changelog 표시
- 크롤러 성능 30% 향상
- 메모리 누수 문제 해결
```

#### ❌ 피해야 할 형식

```markdown
# 복잡한 마크다운

## 여러 섹션으로 나누기

**너무 많은 강조**

- 너무 길고 장황한 설명으로 한 줄에 여러 내용을 담으려고 하면 읽기 어려워집니다
```

### 4. 실제 작성 예시

#### Major 릴리즈 (v2.0.0)

```markdown
- 전체 UI 리뉴얼
- 다크모드 지원 추가
- 실시간 알림 시스템 구현
- 데이터베이스 마이그레이션
```

#### Minor 릴리즈 (v1.3.0)

```markdown
- 휴가 승인 워크플로우 개선
- 엑셀 내보내기 기능 추가
- 대시보드 위젯 3종 추가
```

#### Patch 릴리즈 (v1.2.1)

```markdown
- 로그인 페이지 레이아웃 깨짐 수정
- 날짜 선택기 버그 해결
- 타입스크립트 에러 수정
```

## 릴리즈 체크리스트

릴리즈 전 확인사항:

- [ ] `package.json`의 버전이 릴리즈 버전과 일치하는가?
- [ ] 빌드가 정상적으로 완료되는가?
- [ ] 타입 체크가 통과하는가? (`pnpm typecheck`)
- [ ] 변경사항이 모두 커밋되었는가?
- [ ] 릴리즈 메시지가 간단명료한가?

## 자동 배포

GitHub Releases를 생성하면:

1. 앱에서 자동으로 changelog를 불러옴
2. 사용자가 버전 페이지에서 변경사항 확인 가능
3. electron-updater가 새 버전 감지 및 업데이트 제공

---

**핵심**: 한 줄에 하나의 변경사항, `-`로 시작, 간단명료하게!
